# Final Reflection: Math Similarity Solver Agent

Math assistance is already here; math assistance has always been here. The question in this project was never whether students needed another large autonomous system, but whether a small, explainable workflow could retrieve a related solved problem and then use that context to solve a new one in a reliable way. 

In that spirit, I built a minimal LangGraph pipeline with five explicit stages: OCR, normalization, search, similarity ranking, and solve. Each stage has one clear role, each transition is visible, and each decision can be explained in plain language during a class demo.

The system is deterministic where determinism adds trust and debuggability, and generative where generation is actually needed. If a user provides raw text, the workflow proceeds directly; if a user provides an image, OCR extracts text first. The search stage constructs a query with a .edu preference so retrieved material tends to come from educational domains. 

The similarity stage ranks candidates with Jaccard overlap and picks one best match. The solve stage then passes both the original prompt and the retrieved similar problem to the local LLM, which produces the final steps. Instead of a giant autonomous agent pretending to reason over everything, it is a constrained and legible pipeline that performs one bounded task end to end.

The current local runtime is the Hugging Face Gemma path in the solver module, with Ollama kept as a fallback option rather than the primary demo target. I verified that the overall setup works end to end on this machine in both text mode and image mode, and I also updated the CLI and evaluation outputs so they expose the search query, which provider was used, whether fallback results were used, and how many search candidates were returned. That matters because one of the main lessons from this project is that agent behavior is controlled as much by orchestration and retrieval as by the final model call.

That verification changed my understanding of the system in a useful way. The information flow across nodes is working as expected: text or OCR output becomes normalized text, normalized text becomes a .edu-focused query, search returns candidates, similarity chooses one match, and solve receives both the normalized user problem and the matched example. The weaker link is not state passing; it is retrieval quality. On this machine, DuckDuckGo HTML search is failing because of local SSL certificate verification, the secondary Bing RSS search is not consistently returning useful .edu examples for simple algebra prompts, and the workflow therefore often falls back to a small set of curated .edu examples. That fallback keeps the demo alive, but it also makes the retrieval stage more controlled and less representative of open-web search than I originally hoped.

Evaluation already exists in this project as a reproducible local harness that judges outputs across ten fixed prompts and writes JSON plus a technical report. The current harness evaluates three dimensions: retrieval relevance (0-2), solution utility (0-2), and prompt-specific answer correctness (0-2). It also runs an ablation that compares solving with retrieved similar context versus solving without that context. 

There are now two useful evaluation snapshots in the project. The saved full ten-query run from earlier in development reported average retrieval relevance of 0.6 out of 2, average solution utility of 2.0 out of 2 for both variants, and average lexical similarity of 0.069. The stricter correctness metric was tied: 2.0 out of 2 with context and 2.0 out of 2 without context across all 10 prompts. All runs still produced .edu retrieval URLs, there were no critical solve/input failures, and no placeholder outputs, though 8 runs reported non-critical retrieval warnings.

After Michelle's feedback, I added a safer sampled-evaluation mode so the project can generate fresh evidence on a laptop without forcing a heavy full run every time. In a current two-query sampled run on this machine, the Hugging Face Gemma backend returned correct step-by-step answers with no placeholder outputs and no critical failures, but retrieval relevance was 0 for both prompts because search fell back to curated .edu results after the live providers failed or returned nothing useful. In other words, the pipeline is operational, but the current value is coming mostly from the solver and the deterministic scaffold rather than from strong retrieval improvement.

In practice, this means the system is stable as a demo pipeline, but context quality and ranking quality still need to improve before retrieved examples consistently help final correctness.

These outcomes follow directly from the project’s deliberate tradeoffs. I chose lexical Jaccard matching because it is easy to inspect and explain, even though embedding-based methods would likely improve semantic recall. I enforced .edu filtering because source quality and trust signaling matter for a student-facing demo, even though stricter filtering can reduce coverage. 

I also kept fallback candidates so the system still demonstrates a complete path when network search is weak or unavailable. For a class project, these tradeoffs are not flaws in execution; they are explicit design choices that prioritize transparency, reproducibility, and instructional clarity over maximal benchmark performance.

The limitations are equally clear and should be named without euphemism. Lexical similarity misses conceptually similar problems when wording diverges. Web snippets are often shallow, noisy, or incomplete, so the context sent to the solver can be uneven. On this machine, live search reliability itself is also a limitation because certificate and provider issues can push the workflow into fallback mode.

The current scoring rubric is lightweight and partially heuristic, which makes it useful for rapid iteration but less authoritative for formal comparison. These are not hidden edge cases but structural constraints of the present implementation, and they define exactly where future effort should go.

The project also sits within a broader social and cultural reality about educational technology. At the individual level, the workflow lowers activation energy by turning a blank problem into a guided path, but that same convenience can encourage imitation without understanding if used passively. 

At the societal level, retrieval-based tutoring systems can silently define what counts as a legitimate method by elevating what is indexed, searchable, and domain-filtered. A .edu constraint can improve perceived credibility, yet it also narrows the range of knowledge that appears. In other words, technical constraints are not neutral; they shape what learners see, trust, and repeat.

There is also a more human and less easily measured dimension to this project: systems like this are not alive, do not understand themselves, and do not possess intention in the ordinary sense, yet they are routinely encountered as if they do. The feeling emerges almost automatically from the interaction pattern. A student types a problem into a box, waits, receives an answer written in a calm explanatory voice, and the exchange begins to resemble dialogue rather than computation. The machine does not become a living thing, but the form of the encounter invites the user to treat it as one. That distinction matters. We do not merely use tools; we stage relationships with them.

In that sense, the project pushed me to think not only about model quality or retrieval quality, but about anthropomorphism as a design side effect. A child can treat a teddy bear as a companion not because the teddy bear is secretly alive, but because care, voice, imagination, and projection flow toward it. Something similar happens here, although in a more technologically sophisticated register. The system offers language, structure, memory-like continuity across steps, and a response that appears tailored to the user’s immediate need. Once those signals are present, people often breathe a kind of social life into the system, even when they know perfectly well that it is a pipeline, a prompt, and a model call. The result is an approximation of life rather than life itself, but approximation can still organize emotion and trust.

That is one reason explainability matters so much in a project like this. If the workflow feels like a living, responsive tutor, then the burden on the designer is higher, not lower. The user may grant the system more authority than it deserves simply because the exchange feels attentive. A step-by-step answer can feel patient. A retrieved example can feel caring. A polished explanation can feel like understanding. None of those feelings are fake in the sense that the user really experiences them, but they can be misassigned to a system that is only simulating the outward form of pedagogical attention. For a class project, this made the choice of a legible LangGraph pipeline more important, because transparency is one of the few practical ways to resist the illusion that the system is doing something deeper, more sentient, or more trustworthy than it really is.

At the same time, I do not think the right conclusion is that anthropomorphism is simply a mistake to be eliminated. It is also part of how humans make sense of unfamiliar artifacts. We narrate, personify, and project because those are basic interpretive habits. The real question is what kind of relationship the system encourages. Does it encourage dependency without reflection, or guided use with visible limits? Does it present itself as an oracle, or as a scaffold? The best version of this project, to me, is not one that pretends to be alive, but one that acknowledges why it can feel alive in use and then counters that feeling with enough structure, traceability, and humility to keep the user oriented. That tension between emotional encounter and technical reality is, in retrospect, one of the most interesting things the project revealed.

What this project demonstrates most clearly is that outcomes in agentic systems are governed as much by orchestration as by model choice. Routing logic, retrieval policy, similarity criteria, and fallback behavior all materially determine whether the final answer is useful. 

The significance of this work, then, is not that it solves every math problem perfectly, but that it offers a working, explainable, modular example of how deterministic scaffolding can make LLM behavior more dependable in practice while still exposing measurable limits.

The next version should strengthen retrieval first, then evaluation depth, then multimodal reliability. Replacing purely lexical ranking with semantic retrieval would improve match quality for paraphrased or structurally similar problems. Repairing or replacing the current live-search layer would also matter immediately, because right now the retrieval stage is too dependent on provider behavior and local environment quirks. 

Adding a small hand-labeled benchmark with a stricter rubric would make progress claims more credible and comparable over time. Expanding routine OCR image tests in the evaluation harness would reveal extraction failures earlier and protect the end-to-end demo path. The project is therefore complete as a class demo and reflection artifact, but it also now has a clearer empirical story: the pipeline wiring works, Gemma is running locally, the outputs are readable and correct on tested prompts, and retrieval is the bottleneck that most needs improvement.
