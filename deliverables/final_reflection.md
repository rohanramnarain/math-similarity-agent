# Final Reflection: Math Similarity Solver Agent

Math assistance is already here; math assistance has always been here. The question in this project was never whether students needed another large autonomous system, but whether a small, explainable workflow could retrieve a related solved problem and then use that context to solve a new one in a reliable way. The point of this project is also that math questions that are published on the internet on educational websites change over time and advances in math and changes in common curricula are also reflected in the constant "drift" of online math problems. 

In that spirit, I built a minimal LangGraph pipeline with five explicit stages: OCR, normalization, search, similarity ranking, and solve. Each stage has one clear role, each transition is visible, and each decision can be explained in plain language during a class demo.

The system is deterministic where determinism adds trust and debuggability, and generative where generation is actually needed. If a user provides raw text, the workflow proceeds directly; if a user provides an image, OCR extracts text first. The search stage constructs a query with a .edu preference so retrieved material tends to come from educational domains. 

The similarity stage ranks candidates with Jaccard overlap and picks one best match. The solve stage then passes both the original prompt and the retrieved similar problem to the local LLM, which produces the final steps. Instead of a giant autonomous agent pretending to reason over everything, it is a constrained and legible pipeline that performs one bounded task end to end.

The current local runtime is the Hugging Face Gemma path in the solver module, with Ollama kept as a fallback option rather than the primary demo target. Based on the saved evaluation artifacts in this repo, the setup runs end to end in text mode, and I also updated the CLI and evaluation outputs so they expose the search query, which provider was used, whether fallback results were used, and how many search candidates were returned. That matters because one of the main lessons from this project is that agent behavior is controlled as much by orchestration and retrieval as by the final model call.

A current smoke check also shows that runtime behavior depends on the active Python environment. In one local environment, Hugging Face Gemma loading can fail and produce a placeholder solution (for example, when model compatibility or token/access is not fully configured). So the pipeline wiring is intact, but reliable solve quality still depends on environment readiness.

That verification changed my understanding of the system in a useful way. The information flow across nodes is working as expected: text or OCR output becomes normalized text, normalized text becomes a .edu-focused query, search returns candidates, similarity chooses one match, and solve receives both the normalized user problem and the matched example. 

On this machine, DuckDuckGo HTML search still fails at times because of local SSL certificate verification, so I changed the live-search order to try Bing HTML first, then Bing RSS, then DuckDuckGo HTML, and only then use curated fallback examples. In current checks, Bing RSS is the provider that most consistently returns usable .edu results. For example, it pulled Paul's Online Math Notes at https://tutorial.math.lamar.edu/Classes/Alg/Alg.aspx for simple algebra prompts and MIT OpenCourseWare at https://ocw.mit.edu/courses/res-18-001-calculus-fall-2023/pages/open-textbook/ for derivative prompts. Bing HTML can still return no .edu candidates for some queries, but the reordered stack reduces hard failures and keeps retrieval live more often before fallback is needed.

Evaluation already exists in this project as a reproducible local harness that judges outputs across ten fixed prompts and writes JSON plus a technical report. The current harness evaluates three dimensions: retrieval relevance (0-2), solution utility (0-2), and prompt-specific answer correctness (0-2). It also runs an ablation that compares solving with retrieved similar context versus solving without that context. 

There are now two useful evaluation snapshots in the project. The saved full ten-query run from earlier in development reported average retrieval relevance of 0.6 out of 2, average solution utility of 2.0 out of 2 for both variants, and average lexical similarity of 0.069. The stricter correctness metric was tied: 2.0 out of 2 with context and 2.0 out of 2 without context across all 10 prompts. All runs still produced .edu retrieval URLs, there were no critical solve/input failures, and no placeholder outputs, though 8 runs reported non-critical retrieval warnings.

After Professor Michelle's feedback, I added a safer sampled-evaluation mode so the project can generate fresh evidence on a laptop without forcing a heavy full run every time. In a current two-query sampled run on this machine, the Hugging Face Gemma backend returned correct step-by-step answers with no placeholder outputs and no critical failures, but retrieval relevance was 0 for both prompts because search fell back to curated .edu results after the live providers failed or returned nothing useful. In other words, the pipeline is operational, but the current value is coming mostly from the solver and the deterministic scaffold rather than from strong retrieval improvement.

In practice, this means the system is stable as a demo pipeline, but context quality and ranking quality still need to improve before retrieved examples consistently help final correctness.

These outcomes follow directly from the project’s deliberate tradeoffs. I chose lexical Jaccard matching because it is easy to inspect and explain, even though embedding-based methods would likely improve semantic recall. I enforced .edu filtering because source quality and trust signaling matter for a student-facing demo, even though stricter filtering can reduce coverage. 

I also kept fallback candidates so the system still demonstrates a complete path when network search is weak or unavailable. For a class project, these tradeoffs are design choices that prioritize transparency over maximal benchmark performance.

The limitations are equally clear and should be named without euphemism. Lexical similarity misses conceptually similar problems when wording diverges. Web snippets are often shallow, noisy, or incomplete, so the context sent to the solver can be uneven. On this machine, live search reliability itself is also a limitation because certificate and provider issues can push the workflow into fallback mode.

The current scoring rubric is lightweight and partially heuristic, which makes it useful for rapid iteration but less authoritative for formal comparison. These are structural constraints of the present implementation, and they define exactly where future effort should go.

The project also sits within a broader social and cultural reality about educational technology. At the individual level, the workflow lowers activation energy by turning a blank problem into a guided path, but that same convenience can encourage imitation without understanding if used passively. 

At the societal level, retrieval-based tutoring systems can silently define what counts as a legitimate method by elevating what is indexed, searchable, and domain-filtered. A .edu constraint can improve perceived credibility, yet it also narrows the range of knowledge that appears. In other words, technical constraints have inescapable biases.

There is also a more human and less easily measured dimension to this project, which is where we venture to the humanities side of everything - I majored in economics, which is technically a social science, so bear with me for a second. Systems like this are not alive, do not understand themselves, and do not possess intention in the ordinary sense, yet they are routinely encountered by society as if they do. The feeling emerges almost automatically from the interaction pattern. A student types a problem into a box, waits, receives an answer written in a calm explanatory voice, and the exchange begins to resemble dialogue rather than computation. Better yet, society teaches children of multiple generations to expect this behavior and learn it fresh out of the womb. The machine does not become a living thing, but the form of the encounter invites the user to treat it as one. 

So when you look at it that way, the project pushed me to think about model quality and retrieval quality, while considering anthropomorphism as a design side effect. A child can treat a teddy bear as a companion even though the teddy bear is inanimate, but because care, voice, imagination, and projection flow toward it, humanity makes the inanimate seem and feel more human to human beings, almost as though humanity is infectious in the eyes of a human. Something similar happens here, although in a more technologically sophisticated register.

The system offers language, structure, memory-like continuity across steps, and a response that appears tailored to the user’s immediate need. Once those signals are present, people often breathe a kind of social life into the system, even when they know perfectly well that it is a pipeline, a prompt, and a model call. The result is an approximation of life rather than life itself, but approximation can still organize emotion and trust, but perhaps only in the eyes of the emotional and trusting human, such as myself.

That is one reason explainability matters so much in a project like this. If the workflow feels like a living, responsive tutor, then the burden on the designer is higher, not lower. The user may grant the system more authority than it deserves simply because the exchange feels attentive. A step-by-step answer can feel patient. A retrieved example can feel caring. A polished explanation can feel like understanding. None of those feelings are fake in the sense that the user really experiences them, but they can be misassigned to a system that is only simulating the outward form of pedagogical attention. For a class project, this made the choice of a legible LangGraph pipeline more important, because transparency is one of the few practical ways to resist the illusion that the system is doing something deeper, more sentient, or more trustworthy than it really is.

At the same time, I do not think the right conclusion is that anthropomorphism is simply a mistake to be eliminated. It is also part of how humans make sense of unfamiliar artifacts. We narrate, personify, and project because those are basic interpretive habits. The real question is what kind of relationship the system encourages. Does it encourage dependency without reflection, or guided use with visible limits? Does it present itself as an oracle, or as a scaffold? The best version of this project, to me, is one that acknowledges why it can feel alive in use and then counters that feeling with enough structure, traceability, and humility to keep the user oriented. That tension between emotional encounter and technical reality is, in retrospect, one of the most interesting things the project revealed.

And then the fascinating question is, aren't we all using technology to simply act as an "advanced mirror" of what we see in ourselves, what we hate in ourselves, what we love about ourselves, the future dream we have for ourselves, the polished retconned version of the past that we imagine of ourselves, and aren't our interactions with other humans also based on using them as an "advanced mirror" where we talk to other humans and pretend to be interested in their lives, but ultimately we are looking to other humans for validation of our own private thoughts and morals and our view of "the way the world works" - so isn't the natural progression of technology meant to function as this advanced mirror reflecting humanity and its flaws back to us in new and creative ways?

One transparency point is important here: parts of this project were developed with the help of coding agents and LLM-assisted tooling, but the delivered artifact is not itself a fully autonomous agentic system. It is a staged workflow with explicit nodes and bounded behavior. That distinction reinforces the relevance of the broader ethical and social analysis done here, and what kinds of knowledge are made more visible or less visible to users.

What surprised me most at the final showcase was how a technically simple tool could still trigger a deep conversation about education and AI at a systems level. After the presentation, a professor from Columbia spoke with me about the project, and that conversation expanded into a three-way discussion with Professor Michelle and me about how AI may transform teaching structures without eliminating the core human act of teaching. We ended up with a provocative shared view: pedagogy is likely to survive and adapt, but some current institutional forms around education may not.

What this project demonstrates most clearly is that outcomes in agentic systems are governed as much by orchestration as by model choice. Routing logic, retrieval policy, similarity criteria, and fallback behavior all materially determine whether the final answer is useful. 

The significance of this work, then, is that it offers a working, explainable, modular example of how deterministic scaffolding can make LLM behavior more dependable in practice while still exposing measurable limits.

The next version should strengthen retrieval first, then evaluation depth, then multimodal reliability. Replacing purely lexical ranking with semantic retrieval would improve match quality for paraphrased or structurally similar problems. Repairing or replacing the current live-search layer would also matter immediately, because right now the retrieval stage is too dependent on provider behavior and local environment quirks. 

Adding a small hand-labeled benchmark with a stricter rubric would make progress claims more credible and comparable over time. Expanding routine OCR image tests in the evaluation harness would reveal extraction failures earlier and protect the end-to-end demo path. The project is therefore complete as a class demo and reflection artifact, but it also now has a clearer empirical story: the pipeline wiring works, Gemma is running locally, the outputs are readable and correct on tested prompts, and retrieval is the bottleneck that most needs improvement.
